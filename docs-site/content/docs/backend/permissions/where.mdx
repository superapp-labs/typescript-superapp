Where clauses inject SQL `WHERE` conditions into every query so users only see and touch rows they're allowed to. They work the same way as [Drizzle ORM filters](https://orm.drizzle.team/docs/select#filtering) — if you know Drizzle, you already know how this works.

## Complete Example

One permission that covers all four operations on an `orders` table:

```typescript
permissions: {
  manage_orders: {
    table: 'main.orders',
    roles: ['member'],

    // READ — only orders from the user's organization
    select: {
      columns: ['id', 'amount', 'status', 'customer_id', 'created_at'],
      where: {
        organization_id: { $in: '$user.org_ids' },
        status: { $ne: 'deleted' },
      },
    },

    // CREATE — no where clause needed (new rows)
    insert: {
      columns: ['amount', 'status', 'customer_id'],
      validate: { amount: { $gte: 0 } },
      default: { status: 'draft' },
      overwrite: {
        created_by: '$user.id',
        organization_id: '$user.current_org_id',
      },
    },

    // UPDATE — can only update orders in your org that aren't completed
    update: {
      columns: ['amount', 'status'],
      where: {
        organization_id: { $in: '$user.org_ids' },
        status: { $ne: 'completed' },
      },
      validate: { amount: { $gte: 0 } },
      overwrite: { updated_by: '$user.id' },
    },

    // DELETE — can only delete drafts in your org
    delete: {
      where: {
        organization_id: { $in: '$user.org_ids' },
        status: { $eq: 'draft' },
      },
    },
  },
}
```

**What the engine generates for a `select` query:**

```sql
SELECT id, amount, status, customer_id, created_at
FROM main.orders
WHERE organization_id IN ('org_1', 'org_2')
  AND status != 'deleted'
```

**For an `update`:**

```sql
UPDATE main.orders
SET amount = 500, updated_by = 'usr_123'
WHERE organization_id IN ('org_1', 'org_2')
  AND status != 'completed'
  AND id = 'order_42'
```

**For a `delete`:**

```sql
DELETE FROM main.orders
WHERE organization_id IN ('org_1', 'org_2')
  AND status = 'draft'
  AND id = 'order_42'
```

The user never sees or bypasses these conditions — they're injected server-side on every request.

## Operators

Where clauses use MongoDB-style operators. These map directly to SQL:

```typescript
// Exact match
where: { status: { $eq: 'active' } }
// → WHERE status = 'active'

// Not equal
where: { status: { $ne: 'deleted' } }
// → WHERE status != 'deleted'

// List membership
where: { status: { $in: ['active', 'pending'] } }
// → WHERE status IN ('active', 'pending')

// Range
where: { amount: { $gte: 0, $lte: 50_000 } }
// → WHERE amount >= 0 AND amount <= 50000

// Multiple conditions are AND'd together
where: {
  organization_id: { $eq: '$user.current_org_id' },
  status: { $ne: 'deleted' },
}
// → WHERE organization_id = 'org_1' AND status != 'deleted'
```

See [Operators](/docs/backend/permissions/operators) for the full list (`$eq`, `$ne`, `$gt`, `$gte`, `$lt`, `$lte`, `$in`, `$nin`, `$like`, `$ilike`).

## Session Variables

Any property on the user session (returned by [`resolveSession`](/docs/backend/auth)) is available as a `$user.*` token:

```typescript
where: { customer_id: { $eq: '$user.id' } }
// → WHERE customer_id = 'usr_123'

where: { organization_id: { $in: '$user.org_ids' } }
// If $user.org_ids = ['org_1', 'org_2']
// → WHERE organization_id IN ('org_1', 'org_2')
```

## Logical Operators

Combine conditions with `$or`, `$and`, and `$not`:

```typescript
where: {
  $or: [
    { customer_id: { $eq: '$user.id' } },
    { assigned_to: { $eq: '$user.id' } },
  ],
}
// → WHERE (customer_id = 'usr_123' OR assigned_to = 'usr_123')
```

```typescript
where: {
  status: { $ne: 'deleted' },
  $or: [
    { customer_id: { $eq: '$user.id' } },
    { organization_id: { $in: '$user.org_ids' } },
  ],
}
// → WHERE status != 'deleted' AND (customer_id = 'usr_123' OR organization_id IN ('org_1', 'org_2'))
```

## Relationship Traversal

Filter through foreign key relationships by nesting objects. The engine follows the FK path in your schema:

```typescript
select: {
  columns: '*',
  where: {
    organization: {
      members: {
        user_id: { $eq: '$user.id' },
        role: { $in: ['owner', 'admin'] },
      },
    },
  },
}
```

**Generated SQL:**

```sql
SELECT * FROM main.orders
WHERE organization_id IN (
  SELECT id FROM main.organizations
  WHERE id IN (
    SELECT organization_id FROM main.members
    WHERE user_id = 'usr_123'
      AND role IN ('owner', 'admin')
  )
)
```

The engine resolves nested keys by inspecting FK relationships:

```text
main.orders
  → organization (FK: orders.organization_id → organizations.id)
    → members (FK: members.organization_id → organizations.id)
      → user_id = $user.id, role IN ('owner', 'admin')
```

You can mix direct column conditions with FK traversal:

```typescript
where: {
  status: { $ne: 'deleted' },
  organization: {
    members: { user_id: { $eq: '$user.id' } },
  },
}
```

The `maxFilterDepth` setting (default: `5`) controls how many FK hops are allowed. Exceeding it returns `400 Bad Request`.

## Permission Merging

When a user matches multiple permissions for the same table and operation, where clauses are merged with OR:

```typescript
// Permission 1: view own orders
select: { where: { customer_id: { $eq: '$user.id' } } }

// Permission 2: view org orders
select: { where: { organization_id: { $in: '$user.org_ids' } } }

// Merged:
// → WHERE (customer_id = 'usr_123') OR (organization_id IN ('org_1', 'org_2'))
```
